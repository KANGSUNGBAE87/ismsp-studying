# 결함 판단문 모드 재설계 기획안

ISMS-P "결함사항 판단문 공부" 모드(현재 첫 번째 모드)의 풀이 방식과 문장 제시
방식을 바꾸고, 오답 해설을 강화하기 위한 새 퀴즈/학습 방식 기획안.

- 대상 모드: `결함사항 판단문 공부` (engine: `createSession` / `createDefectJudgmentSession`)
- 작성 범위: 기획만(코드 미수정). 기존 두 모드(`결함사항 기준맞추기`, `확인사항 기준맞추기`)는 유지.
- 산출 목적: 인지적으로 더 강한 풀이 방식 + 새 문장 제시 + 해설 강화 아이디어 도출.

---

## 1. 현재 동작 요약 (분석 결과)

### 1.1 엔진 동작 (`src/core/quizEngine.js`)

- `createSession`: `falsePool`에서 3개 + `truePool`에서 2개를 뽑아 5지선다로 섞음.
- 학습자는 "옳은 판단문 2개"를 고름. `evaluateAnswer`는 선택 수가 정확히 2개이고
  두 개 모두 `isCorrect: true`일 때만 정답.
- 보기 텍스트는 워크북 고정 문장: `「결함사례 본문」 X.Y.Z 기준명 결함에 해당한다.`
- 오답을 누르면 `explainIncorrectOption()`이 같은 결함사례의 진짜 정답(원문 매핑)을
  찾아 해설을 보기 아래에 표시. 매칭 등급은 `exact`(동일 결함사례) / `similar`(유사도
  >= 0.46) / `none`(못 찾음).

### 1.2 데이터 자산 (`public/data/ismsp-defect-bank.json`)

직접 확인한 실제 구조와 수치:

- `criteria`: 101개. 각 항목에 `code/name/domain/groupCode/groupName/requirement/
  checkItemCount`. **101개 모두 `requirement` 텍스트 있음**(해설 강화의 핵심 자산).
  단 `hint` 필드는 없음.
- `truePool`: 381개. `{statement, criteriaCode(정답), criteriaName, defectCase, isCorrect:true, sourceRow}`
- `falsePool`: 1143개. 같은 `defectCase`에 **틀린** `criteriaCode`를 붙여 만든 오답.
- `defectCasePool`: 381개. 결함사례 원문 + 정답 기준.
- `checkItemPool`: 496개. 확인사항 + 정답 기준(가상자산 배지 56개).
- `similarCriteriaByCode`: 101개 코드 전부, 각 **정확히 12개**의 유사 기준 리스트.

### 1.3 결정적 발견: truePool/falsePool의 묶임 구조

요구사항에 적힌 "같은 defectCase가 true 1개 + false 여러개"는 **부분적으로만 맞다.**
직접 검증한 실제 분포:

- 고유 true 결함사례: 381개 / 고유 false 결함사례: 762개.
- **true 결함사례당 false는 1~2개뿐** (380개는 2개, 1개는 1개). 평균 2.0.
- false 1143행 중 **761행만** 동일 결함사례의 true 행과 정확히 매칭된다.
- 나머지 **382행은 "고아(orphan) false"** 로, 동일 결함사례 true가 없다.
  - 고아 false는 결함사례 본문을 살짝 바꾼 변형(예: "장기간 보고 미수행" →
    "분기 1회 명시했으나 당월 보고 미수행")이고, 붙은 코드가 **원래 정답 코드와
    같은 경우도 있음**. 즉 단순히 코드만 틀린 게 아니라 본문 자체가 변형된 케이스.

함의:

- 현 `explainIncorrectOption`은 **exact 매칭이 약 761/1143행에서만** 깔끔히 동작하고,
  나머지는 similar(유사도) 폴백 또는 none으로 떨어진다. 해설 강화 시 이 구멍을 메우는
  게 가장 큰 가치(아래 "해설 강화 공통 전략" 참고).
- "옳은 2개 고르기"가 인지적으로 약한 이유: 5개 중 true 2개는 항상 **서로 다른
  결함사례**라서, 학습자가 결함사례 본문을 안 읽고 "기준명-문장 톤"만으로 찍어도
  맞히기 쉬운 구조다. 결함사례↔기준 매핑을 진짜로 따지지 않아도 통과 가능.

### 1.4 난이도 설계용 도메인 분포

- truePool/defectCasePool: 도메인1=60, 도메인2=232, 도메인3=89.
- checkItemPool: 도메인1=60, 도메인2=285, 도메인3=151.
- truePool은 101개 기준 전부 커버. similar 리스트는 코드당 12개 고정.

## 2. 해설 강화 공통 전략 (모든 아이디어에 공통 적용)

현 `explainIncorrectOption` / `explainCriterionChoice`를 재사용·확장하는 공통 토대.
아이디어별 해설은 이 토대 위에서 변형한다.

1. **정답 근거를 항상 기준 요구사항으로 못박기**: 101개 `requirement`가 전부 있으므로,
   "왜 이 결함사례가 X.Y.Z인가"를 `criteria[code].requirement`로 1줄 인용. (이미 부분 구현)
2. **오답 기준과의 대비 강화**: 선택한 틀린 기준의 `requirement`도 같이 인용해
   "선택한 기준은 ~에 관한 것이라 결함사례 핵심과 다르다"를 명시. (이미 부분 구현)
3. **고아 false 구멍 메우기**: exact/similar 둘 다 실패할 때 현재는 "원문 못 찾음"으로
   끝난다. 대신 **같은 group/domain 안에서 키워드 겹침이 가장 큰 정답 후보**를 보조
   근거로 제시하거나, 최소한 "이 결함사례의 핵심 키워드(예: 암호정책, 접근권한)는 어느
   기준군의 관심사인지"를 group 단위로 설명. → `similarCriteriaByCode` + `groupName` 활용.
4. **해설 톤 통일**: 단정형 1문장 요약 + 근거 불릿. 예) `틀린 판단입니다. 이 결함사례는
   2.7.1 암호정책 적용이 아니라 2.6.1 네트워크 접근 결함에 해당합니다.` + 근거 2~3줄.
5. **"맞게 고른 것"에도 칭찬+근거**: 정답 보기를 골랐을 때도 왜 맞는지 1줄을 보여줘
   정적 강화(positive reinforcement)와 기준 암기를 동시에 노린다.

이 토대는 새 `explainJudgmentVerdict(...)` 같은 얇은 래퍼로 묶되, 내부는 기존
`findExactOriginal` / `findSimilarOriginal` / `getCriteria` / `truncateText`를 그대로 호출.

## 3. 새 퀴즈/학습 방식 아이디어 (5개)

---

### 아이디어 A. 오류 사냥 — "틀린 판단 2개를 골라내라" (요구사항 예시의 반전)

- **컨셉 한 줄**: 5개 판단문 중 잘못된 판단(오류) 2개를 적발한다.
- **훈련 인지과정**: 오류 탐지(error detection) + 기준 변별. "맞는 걸 고르기"보다
  "틀린 걸 적발"이 심사원의 실제 업무(결함 판정)와 일치하고, 본문을 안 읽으면 못 푼다.
- **풀이 방식**: 보기 5개(true 3 + false 2로 비율을 **뒤집음**), 선택 2개, 정확히
  false 2개를 모두 고를 때만 정답. (현 `truePool` 3 + `falsePool` 2로 슬라이스만 변경)
- **화면 문구 예시**:
  - 프롬프트: `다음 중 심사원의 판단으로 잘못된 것 2개를 고르시오. (틀린 판단 = 결함 기준 오적용)`
  - 보기 텍스트(유지): `「분기 보고를 장기간 미수행한 경우」 1.1.2 최고책임자의 지정 결함에 해당한다.`
  - 상태/피드백: `정답입니다. 2개의 오적용을 정확히 적발했습니다.` / `틀린 판단을 1개 놓쳤습니다.`
    / `맞는 판단을 오류로 표시했습니다.`
  - 해설 톤: 골라낸 false에 대해 "이 결함사례의 진짜 기준은 ~"; 놓친 false에 대해
    "이것도 오적용입니다: ~가 아니라 ~".
- **데이터 실현성**: **기존 데이터로 즉시 가능.** `createSession`의 슬라이스 비율과
  판정 규칙만 반전. 추가 데이터 0.
- **해설**: `explainIncorrectOption`을 **선택한 false뿐 아니라 놓친 false에도** 호출하도록
  확장. 정답(true) 보기에는 §2-5 근거 1줄.
- **학습효과/난이도**: 학습효과 중상(현 모드보다 본문 의존도↑), 난이도 중. **구현 복잡도: 하.**

### 아이디어 B. 단일 결함사례 정답 판정 — "이 한 줄 판단, 맞나 틀리나" (O/X + 근거)

- **컨셉 한 줄**: 하나의 결함사례에 대한 단일 판단문을 O/X로 판정하고, X면 올바른 기준을 잇는다.
- **훈련 인지과정**: 근거 추론 + 단일 매핑 검증. 5지선다의 "소거법 찍기"를 제거해
  매 문장을 실제로 판정하게 만든다.
- **풀이 방식**: 보기 2개(O/X), 선택 1개. 제시 문장이 true면 정답=O, false면 정답=X.
  X 정답 후, 후속 단계로 "그럼 올바른 기준은?"을 `similarCriteriaByCode`로 5지선다 1문제
  추가(2단계 문항). 단순형은 O/X만으로도 성립.
- **화면 문구 예시**:
  - 프롬프트: `다음 결함사례에 대한 심사원의 판단입니다. 옳으면 O, 틀리면 X.`
  - 본문 제시: 결함사례 원문 + 판단문을 **분리 제시**(아래 아이디어 D의 문장 생성 방식과 결합 가능).
  - 상태/피드백: `정답입니다(O). 이 판단은 옳습니다.` / `정답은 X입니다. 이 결함사례는 ~ 기준입니다.`
  - 해설 톤: X일 때 "오적용된 기준 vs 올바른 기준" 요구사항 대비.
- **데이터 실현성**: **즉시 가능.** true/false 각 풀에서 1행 추출. 2단계 기준 선택은
  기존 `createDefectCriterionSession` 로직 재사용.
- **해설**: `evaluateSingleAnswer` + `explainIncorrectOption`(X 판정 시) 재사용. 2단계는
  `explainDefectCriterionAnswer` 재사용.
- **학습효과/난이도**: 학습효과 상(문장당 판정 강제), 난이도 중. **구현 복잡도: 하~중**(O/X만 하).

### 아이디어 C. 한 결함사례 · 같은 본문 5개 기준 — "어느 기준이 맞나" (본문 고정, 기준만 변별)

- **컨셉 한 줄**: **동일 결함사례 본문**에 5개의 서로 다른 기준을 붙인 판단문을 보여주고,
  옳은 판단 1개를 고른다.
- **훈련 인지과정**: 순수 기준 변별(criterion discrimination). 본문이 고정되므로 학습자는
  오직 "이 결함이 어느 기준이냐"에만 집중하게 된다 — 현 모드의 가장 큰 약점(서로 다른
  결함사례 5개라 본문 안 읽어도 됨)을 정면으로 교정.
- **풀이 방식**: 보기 5개(모두 같은 결함사례, 기준만 다름), 선택 1개. 정답=true 매핑 1개.
- **화면 문구 예시**:
  - 프롬프트: `다음 결함사례에 대해 올바른 결함 판단 1개를 고르시오.`
  - 보기(같은 본문, 기준만 변형): `「…장기간 보고 미수행…」 1.1.1 경영진의 참여 결함이다.` /
    `「…장기간 보고 미수행…」 1.1.2 최고책임자의 지정 결함이다.` …
  - 상태/피드백: `정답입니다. 1.1.1 경영진의 참여.` / `오답입니다. 정답은 1.1.1입니다.`
  - 해설 톤: 고른 틀린 기준 vs 정답 기준 요구사항 대비(§2-2).
- **데이터 실현성**: **기존 데이터로 가능하나 생성 로직 약간 필요.** 한 `defectCasePool`
  항목의 정답 코드 + `similarCriteriaByCode`에서 4개를 뽑아 **동일 본문에 기준 문자열만
  치환**해 5개 판단문을 런타임 합성. 보기 문장은 `「본문」 {code} {name} 결함에 해당한다`
  템플릿으로 생성(아이디어 D 문장 생성과 동일 원리).
- **해설**: `explainDefectCriterionAnswer` 거의 그대로(보기 표면만 문장형). 폴백 불필요(정답이
  데이터에 확정).
- **학습효과/난이도**: 학습효과 상(현 모드 약점 직접 교정), 난이도 중상. **구현 복잡도: 중.**

---

### 아이디어 D. 빈칸 채우기 판단문 — "이 결함은 ___ 기준이다" (고정 텍스트 탈피)

- **컨셉 한 줄**: 워크북 고정 문장 대신, 결함사례 + 빈칸 판단 템플릿을 제시하고 빈칸에
  들어갈 기준을 고른다(요구사항 2번 "다른 문장 제시 방식" 직접 충족).
- **훈련 인지과정**: 근거 추론 + 생성형 회상(cued recall). "X.Y.Z 결함에 해당한다"를
  통째로 읽혀주지 않고, 학습자가 직접 채우게 해 회상 강도를 높인다.
- **풀이 방식**: 보기 5개(기준 code+name 또는 빈칸 후보), 선택 1개. 정답=정답 기준.
- **화면 문구 예시**:
  - 프롬프트: `다음 결함사례를 읽고, 빈칸에 들어갈 인증기준을 고르시오.`
  - 본문(생성형 제시): `결함사례: 「…암호정책에 따른 키 길이를 적용하지 않은 경우」`
    + 판단 템플릿: `→ 이 사례는 (   ) 결함에 해당한다.`
  - 보기: `2.7.1 암호정책 적용` / `2.6.1 네트워크 접근` / …
  - 상태/피드백: `정답입니다. 2.7.1 암호정책 적용.` / `오답입니다. 올바른 답은 2.7.1입니다.`
  - 해설 톤: 정답 요구사항 1줄 + 결함사례 핵심 키워드 연결("키 길이/암호정책 → 2.7.1").
- **데이터 실현성**: **즉시 가능.** 사실상 현 `createDefectCriterionSession`의 **문장 제시
  방식만 바꾼 변형**이다. 데이터 추가 없음. "고정 텍스트 탈피" 요구를 가장 싸게 충족.
- **해설**: `explainDefectCriterionAnswer` 그대로 재사용 + §2-3 키워드-기준군 보조 근거.
- **학습효과/난이도**: 학습효과 중상, 난이도 중. **구현 복잡도: 하**(기존 모드 재포장 수준).

### 아이디어 E. 결함 vs 정상 분류 — "이건 결함인가, 함정인가" (오답 본문 직접 제시)

- **컨셉 한 줄**: 결함사례 본문 자체와 함께 "이 본문은 실제로 결함 적용이 타당한가,
  아니면 변형된 함정인가"를 분류. 고아 false(본문 변형) 자산을 적극 활용.
- **훈련 인지과정**: 미묘한 본문 변형 탐지(예: "분기 보고 미수행" vs "당월만 미수행이라
  결함 아님 가능성"). 실제 심사에서 결함/비결함 경계 판단력을 훈련.
- **풀이 방식**: 보기 2~3개(`결함 맞음` / `기준 오적용` / `본문상 결함 단정 불가`), 선택 1개.
  - true 본문 → `결함 맞음`
  - false(코드 오적용) → `기준 오적용`
  - 고아 false(본문 변형) → `본문상 결함 단정 불가` 또는 `기준 오적용`
- **화면 문구 예시**:
  - 프롬프트: `다음 판단을 분류하시오.`
  - 보기: `① 결함 적용이 타당함  ② 기준을 잘못 적용함  ③ 본문만으로는 결함으로 단정하기 어려움`
  - 상태/피드백: `정답입니다. 기준 오적용 사례입니다.` / `정답은 ②입니다. 1.1.2가 아니라 1.1.1입니다.`
  - 해설 톤: 변형 포인트를 강조("원문은 '장기간 미수행'인데 여기서는 '당월만 미수행'으로
    바뀌어 결함 단정 근거가 약해졌습니다").
- **데이터 실현성**: **부분 가능, 라벨 신뢰도 주의.** true/false/고아 false 구분은 데이터로
  가능하나, "본문상 단정 불가" 라벨은 §1.3의 고아 false가 **항상 비결함은 아님**(코드만
  같은 변형도 있음)이라 자동 라벨이 부정확할 수 있다. 정확히 하려면 고아 false 382건에
  대한 **소량 수기 라벨링(결함/비결함)** 이 필요.
- **해설**: `explainIncorrectOption` 확장 + 변형 비교는 `findSimilarOriginal`로 가장 가까운
  원문을 찾아 diff 강조.
- **학습효과/난이도**: 학습효과 상(가장 실전적), 난이도 상. **구현 복잡도: 중~상**(수기 라벨 필요).

## 5. 추천 (1~2개) 와 이유

### 1순위: 아이디어 A — "틀린 판단 2개를 골라내라"

- 사용자가 직접 예시로 든 방향이면서, **인지적으로 현 모드의 핵심 약점을 정확히 교정**한다.
  맞는 걸 고르는 건 톤만 봐도 찍히지만, **틀린 걸 적발하려면 결함사례 본문↔기준을 실제로
  대조**해야 한다. 이게 ISMS-P 심사원의 실제 업무(결함 판정)와 가장 가깝다.
- **기존 데이터·UI 구조를 거의 그대로 재사용**한다. true/false 슬라이스 비율과 판정 규칙만
  뒤집고, 멀티셀렉트 UI는 현재 그대로다. 추가 데이터 0, 구현 복잡도 하.
- 해설을 가장 자연스럽게 강화할 수 있다: 선택한 false + **놓친 false**에 대해
  `explainIncorrectOption`을 호출해 "왜 이게 오적용인지"를 2개까지 보여주므로, 한 문항에서
  더 많은 기준 대비 학습이 일어난다.

### 2순위: 아이디어 C — "같은 결함사례, 어느 기준이 맞나"

- A가 "오류 적발"을 훈련한다면, C는 **순수 기준 변별**을 훈련해 상호 보완적이다. 같은 본문에
  기준만 5개라 본문을 안 읽고는 절대 못 푼다(현 모드 약점의 정면 교정).
- 데이터는 `defectCasePool` + `similarCriteriaByCode`로 충분하고, 문장 합성 1개만 추가하면
  된다. 해설은 `explainDefectCriterionAnswer`를 거의 그대로 쓴다.

> 권장 로드맵: **A를 현 첫 모드의 기본 풀이 방식으로 교체**(또는 토글 제공)하고, C를
> 네 번째 모드로 추가. 둘 다 §2 해설 강화 전략을 공유.

## 6. 추천안(A, 부가로 C) 변경 개요 — 코드 구조 기준

> 코드는 수정하지 않음. 아래는 구현 시 어떤 함수/파일을 어떻게 바꾸면 되는지의 설계 수준.

### 6.1 데이터 (`public/data/ismsp-defect-bank.json`)

- **아이디어 A: 추가/변경 없음.** 현 `truePool`/`falsePool`로 충분.
- **아이디어 C: 추가 없음.** `defectCasePool` + `similarCriteriaByCode` 재사용. (선택)
  더 자연스러운 본문 합성을 위해 보기 템플릿 문자열만 i18n에 추가.
- (선택, 해설 품질용) 고아 false 382건에 대한 `verdict` 수기 라벨은 아이디어 E를 할 때만 필요.

### 6.2 엔진 (`src/core/quizEngine.js`)

- **아이디어 A**:
  - `createSession` 슬라이스 반전: `trueOptions = truePool.slice(i*3, i*3+3)`,
    `falseOptions = falsePool.slice(i*2, i*2+2)`. `normalizeCount`의 capacity 계산도
    true/3·false/2로 대칭 변경.
  - 새 프롬프트 상수 `DEFECT_ERROR_HUNT_PROMPT_KO` 추가, 모드 키 `STUDY_MODES`에
    `DEFECT_ERROR_HUNT` 추가(또는 기존 모드에 `variant` 플래그).
  - `evaluateAnswer`는 정답 집합을 "isCorrect=false인 보기들"로 보도록 변형한 새 평가
    함수 `evaluateErrorHunt(question, selected)` 추가(현 함수는 true 기준이라 그대로 두고
    분기). 정답 수=2 고정.
  - 해설: `explainIncorrectOption`을 **선택된 false + 정답이지만 미선택된 false** 양쪽에
    적용하는 래퍼 추가. 정답(true) 보기에는 §2-5용 짧은 정답 근거 함수 추가.
- **아이디어 C**(추가 모드):
  - `createSameCaseCriterionSession(bank, options)` 신설: `defectCasePool`에서 1건,
    `similarCriteriaByCode`로 distractor 4개 → 동일 `defectCase`에 기준 문자열만 치환한
    5개 판단문 보기 생성. 평가는 `evaluateSingleAnswer` 재사용.
  - 해설은 `explainDefectCriterionAnswer` 재사용(보기 표면만 문장형이라 매핑 동일).

### 6.3 UI (`public/app.js`, `public/styles.css`, `src/i18n.js`)

- `public/app.js`:
  - 홈 모드 라우팅에 A(기본 모드 교체 또는 신규 버튼)·C(신규 버튼) 추가.
  - A는 기존 멀티셀렉트 렌더링 재사용(선택 2개 제한 로직 그대로). 피드백 라벨만 "틀린
    판단" 기준으로 교체.
  - 정답 제출 후, 선택 false·미선택 false 양쪽에 해설 블록 렌더(현재는 선택한 오답만
    해설). 정답 보기엔 초록 근거 1줄.
- `src/i18n.js`: A·C용 프롬프트/상태/피드백 ko·en 카피 추가(고정 영어 문구도 함께).

  옵션 스타일 재사용. 모바일 무오버플로 제약 유지.

## 7. 리스크 / 주의

- **고아 false 382건**: 본문 변형이라 exact 매칭이 안 되고, 코드가 정답과 같은 경우도 있어
  자동 "비결함" 단정은 위험. 아이디어 E를 추진할 때만 수기 라벨링으로 해소.
- **해설 폴백 품질**: similar 유사도 0.46 임계 아래로 떨어지면 "원문 못 찾음"이 뜬다.
  §2-3대로 group/요구사항 기반 보조 근거를 넣으면 빈 해설을 줄일 수 있다.
- **모드 교체 vs 추가**: A로 첫 모드를 완전 교체하면 기존 "옳은 2개" 학습 흐름을 좋아하는
  사용자가 잃는 게 있으니, 첫 모드 안에서 "옳은 2개 / 틀린 2개" 토글을 두는 안도 고려.
