# 오답 문제은행 고도화 (오답 합성 · 난이도 보정) 기획안

결함판단문 오답을 현재 ~1,143개에서 **~3,000개**로 늘리되, 단순 증량이 아니라
**검증된 틀림(label integrity) + 보정된 난이도(calibrated difficulty)** 를 자동화하는
것이 목표. 기존 `similarCriteriaByCode`의 표면 글자 유사도 휴리스틱을 **의미적
헷갈림**으로 끌어올린다.

- 작성 범위: 기획만(코드 미수정). 빌드타임 오프라인 파이프라인 설계.
- 런타임 영향 없음: 결과는 기존처럼 `public/data/ismsp-defect-bank.json`에 정적으로 굳힘.
- 선행 기획: `ai/plans/2026-06-15-defect-judgment-redesign.md` (풀이 방식 A·C). 본 기획은
  그 모드들이 소비할 **오답 자산**을 고도화한다.

## 1. 데이터에서 확인한 오답의 "관계 구조" (근거)

`scripts/extract-defect-bank.py`의 `falsePool`(1,143)은 알고리즘 생성물이 아니라
워크북(`결함 테스트_240712v2`, 시트 `(참고) 소스1`)에서 truth=0 행을 **추출**한 것.
즉 추출만으로는 1,143이 천장. 3,000 도달은 **합성(synthesis)** 으로만 가능.

true↔false 직접 비교 결과(검증된 분포):

- **가족 A — 기준 오적용 (코드 관계)**: 761개. 본문 동일, 붙인 기준 코드만 틀림.
  이 중 **273개가 같은 group**(가장 헷갈리는 분류 경계).
- **가족 B — 본문 변형 (의미 관계)**: 고아 false 382개. 진짜 "함정/말장난"이 여기 몰림.
  - matched-case & same-code 본문변형은 0 → 본문변형 트랩은 사실상 전부 고아 382개에 존재.

자산 가용량: `defectCasePool` 381 × `similarCriteriaByCode` 코드당 12 = 최대 4,572 잠재 오답
→ 3,000은 여유 도달(필터링 후에도).

## 2. 트랩 축 분류 (13축 / 2가족) — 합성·검증·라벨의 공통 어휘

### 가족 A — 기준 오적용 (같은 본문, 틀린 기준)
- A1. **같은 group 경계** (예: 2.7.1 ↔ 2.7.2) — 가장 헷갈림
- A2. **같은 domain, 다른 group**
- A3. **다른 domain** (명백히 쉬움 = 하 난이도)

### 가족 B — 본문 변형 (의미 조작) — 데이터에서 실측한 실제 함정
- B1. **주기/빈도** — 매년→반기, 1개월→1주 *(전형적 말장난, 쉬운 축)*
- B2. **범위/대상** — 전체 시스템→일부, 외부자 포함→제외
- B3. **주체** — CISO→CPO, 경영진→실무자
- B4. **시점** — 사전 승인→사후 보고
- B5. **정도(부분 수행/간소화)** — "위험평가를 간소화하고", "1회성으로"
- B6. **면책 조건 추가(정당화)** ★ — "승인을 받아 연기", "구두 승인하에 1회성",
  "타당한 사유로". **결함→비결함으로 뒤집음. 데이터상 최빈·최강 함정.**
- B7. **극성 반전(오히려 더 잘함)** ★ — "더 강화된 패스워드로 운영", "허가하지 않은 경우".
  통제 성공인데 결함처럼 보이게.
- B8. **수단 치환(적절한 대체)** — "외부 타임서버 대신 GPS로 동기화" (목적은 달성)
- B9. **수량/방향 논리 모순** — "권한을 과도하게 *적은* 직원에게 부여"
- B10. **핵심 사유 부정 반전** — "명시되어 있지 않은"(결함)→"명시하고 있지만"(다른 결함)

핵심 통찰: **B6/B7이 진짜 고난도**다. 글자 유사도로는 안 잡히고, "이 조건이 붙으면
결함성이 사라지는가"를 의미적으로 판단해야 함 = AI의 핵심 역할. 수치 말장난(B1)은
오히려 쉬운 축이라 난이도 하/중에 배치.

## 3. 3겹 합성·랭킹 파이프라인 (빌드타임)

> 전부 오프라인 스크립트. 앱 번들에 LLM 키 미포함(전역 규칙 준수). 결과 = 정적 JSON.

### 1겹 — 휴리스틱 사전필터 (유지, 결정적, 비용 0)
- 기존 `build_similar_criteria_by_code` 점수식을 **후보 좁히기/폴백**으로 유지.
- 가족 A 합성의 1차 distractor 후보군 생성에 그대로 사용.

### 2겹 — 의미 유사도 랭킹 업그레이드 (가족 A 고도화)
- char-bigram → **임베딩 의미 유사도** + "심사원이 이 결함을 보면 어떤 기준과 헷갈릴까"를
  LLM에 추론시킨 점수로 교체/가산.
- 산출: 코드별 헷갈림 랭킹을 "표면 글자"에서 "의미적 혼동"으로 재정렬.

### 3겹 — 유형2 LLM 함정 생성 (가족 B, 최종 형태, 1차 포함)
- 입력: `defectCasePool` 1건 + 지정 트랩 축(B1~B10) + 해당 기준 `requirement`.
- 출력(구조화): `{변형문, trapAxis, 바꾼값(before→after), wrongnessReason, correctAnswer}`.
- B6/B7 우선 가중(고난도 비중 확보).
- 생성은 **배치 스크립트 + 동시 API 워커**(throughput). 서브에이전트 아님.

## 4. 검증 아키텍처 (2겹 — 의도적 다중 심판)

### 4.1 빌드 중 자동 검증 — 서브에이전트 배치 심판
- 생성 모델과 **분리된** 심판이 각 후보를 독립 판정(자기검증 편향 제거).
- **배치(50~100건/에이전트)** 로 던져 콜드스타트 분산. 항목 1건당 1에이전트 금지.
- 판정 항목:
  1. 정말 **틀린가** (우연히 여전히 valid한 판단이 아닌가) — label integrity
  2. **그럴듯한가** (명백히 헛소리/비문이 아닌가)
  3. 주장한 **trapAxis가 실제와 일치**하는가
  4. **난이도 1~5** 부여
- 통과 못 한 후보 폐기. 통과분에 `difficulty` 라벨 확정.

### 4.2 난이도 경험 보정 (선택, 권장)
- 약한 "학생" 모델에게 풀려 **속는 비율**로 난이도 재보정. 임의 가중치 대신
  "실제로 모델을 속인 정도"로 난이도 매김 → 휴리스틱 검증.

### 4.3 최종 인간-루프 검증 — GPT 패스 (사용자 수행)
- 최종 문장 완성 후, 사용자가 GPT에 §8 지시문으로 돌려 마지막 라벨 무결성 확인.
- 서브에이전트와 **독립된 제3 심판**이라 다중 방어선 역할.
- 가장 어려운 티어(난이도 4~5)는 사람 표본 검수 병행(전수 아님).

## 7. A·C 모드의 소비 방식

- **A(오류 사냥)**: 큰·어려운·태그된 `falsePool`의 최대 수혜자. `difficulty`로 점진 난이도.
- **C(같은 사례·5기준)**: 2겹 의미 랭킹을 distractor 샘플링에 사용 → "표면 글자"가 아닌
  "의미적 혼동" 기준으로 보기 구성. 런타임 합성이라 falsePool 사전증량 불요하지만
  2겹 랭킹 품질을 직접 상속.

## 8. GPT 최종 검증 지시문 (복붙용)

> 사용자가 ChatGPT/Codex에 그대로 붙여 최종 라벨 무결성을 검증할 때 사용.
> 후보 오답을 §5 스키마 JSON 배열로 함께 제공.

```text
역할: 너는 ISMS-P 인증심사 결함판단 전문 검수자다. 아래 "오답 후보"들이
퀴즈 distractor로 쓰기에 적합한지 한 건씩 엄격히 검증한다. 절대 생성/창작하지 말고
주어진 후보만 판정한다.

입력 각 항목 필드:
- defectCase: 결함사례 본문
- statement: 검증 대상 오답 판단문
- criteriaCode: 이 판단문이 주장하는(틀렸다고 가정된) 기준 코드
- correctAnswer: 출제자가 생각하는 실제 정답 기준 코드
- trapAxis: 주장된 함정 축 (A1~A3 기준오적용 / B1~B10 본문변형)
- wrongnessReason: 왜 틀린지에 대한 출제자 설명

각 항목에 대해 다음을 순서대로 판단한다:
1) [틀림 무결성] 이 statement는 정말로 "잘못된 결함판단"인가?
   - 위험 케이스: 변형/오적용에도 불구하고 statement가 여전히 타당한 판단(=사실상 정답)이
     될 수 있으면 절대 distractor로 못 쓴다. 이 경우 reject.
   - ISMS-P 기준의 requirement 관점에서 근거를 1~2줄로 댄다.
2) [그럴듯함] 명백한 비문/헛소리/현실 불가능 문장이면 reject 또는 revise.
3) [축 일치] 실제 변형/오적용 방식이 trapAxis와 일치하는가? 다르면 올바른 축을 제시.
4) [정답 일치] correctAnswer가 이 결함사례의 실제 정답으로 타당한가? 아니면 올바른 정답 제시.
5) [난이도] 사람 심사원 기준 1~5로 매긴다(1=명백히 틀림, 5=실무자도 속을 함정).
   특히 "면책/정당화 추가(B6)"·"극성 반전(B7)"은 고난도로 본다.

특히 주의: 결함을 "승인/타당한 사유/1회성/더 강화"로 정당화해 비결함으로 보이게 한
변형은, 정말로 결함성이 사라졌는지(=그러면 정답이 'X' 또는 다른 기준이 되는지)를
요구사항 기준으로 엄밀히 따진다.

출력: 입력 순서대로 JSON 배열. 각 원소:
{
  "verdict": "keep" | "revise" | "reject",
  "isGenuinelyWrong": true|false,
  "correctedTrapAxis": "<축 또는 동일>",
  "correctedAnswer": "<코드 또는 동일>",
  "difficulty": 1-5,
  "reason": "<요구사항 근거 포함 2~4줄>",
  "revisedStatement": "<verdict가 revise일 때만, 더 헷갈리되 명백히 틀린 문장>"
}
설명 산문은 JSON 밖에 쓰지 말 것. 확신이 없으면 keep이 아니라 revise/reject로 보수적으로.
```

## 8.5 가족 B 생성을 GPT로 오프로딩 (실행 분업)

Claude 세션 한도로 **생성 에이전트가 반복적으로 중단**되는 문제 발생(2026-06-15 실측:
생성 에이전트 3회 연속 한도 초과, 검증 에이전트는 정상 완료). 따라서 실행 분업:

- **생성 → GPT** (사용자 구동, Claude 한도 우회)
- **검증 → Claude 서브에이전트** (안정적 완료)
- **병합 → `scripts/merge-distractors.py`** (검증 verdict 조인, 미검증분 자동 제외)

워크플로:
1. `scripts/famB-batches/batch-NN.json`을 GPT에 입력으로 제공.
2. 아래 생성 지시문으로 GPT가 후보 생성 → 결과를 `scripts/famB-gen/batch-NN.json`에 저장.
3. Claude가 검증 에이전트로 `scripts/famB-verified/batch-NN.json` 생성.
4. `python3 scripts/merge-distractors.py`로 병합 → `public/data/synth-distractors.json`.

### GPT 생성 지시문 (복붙용)

```text
너는 ISMS-P(정보보호 및 개인정보보호 관리체계 인증) 심사 전문가다. 한국어 스터디 퀴즈용
"본문 변형 함정" 오답 판단문을 생성한다. 아래 JSON 배열의 각 결함사례에 대해 작업한다.
각 항목: defectCase(정답 기준 correctCode에 정확히 매핑되는 진짜 결함), correctCode/
correctName, requirement(판단 기준 근거).

각 사례마다 서로 다른 축의 본문 변형 오답 3개를 만든다. 본문 변형 함정 = 결함사례 본문을
미묘하게 편집해 판단이 "틀리게" 되도록 하되 그럴듯하게 보이게 함. 문장은 "... 결함에
해당한다." 형태로 끝낸다. 고가치 축 B6/B7을 우선(사례당 최소 1개 B6 또는 B7):
- B1 주기/빈도(매년→반기, 1개월→1주) · B2 범위/대상(전체→일부) · B3 주체(CISO→CPO)
- B4 시점(사전→사후) · B5 정도(간소화/1회성/일부만)
- B6 면책 조건 추가(타당한 사유로 승인받아 등 → 실제로는 비결함인데 "결함"이라 주장) ★
- B7 극성 반전(더 강화/허가하지 않음 등 통제 성공인데 결함처럼) ★
- B8 수단 치환(적절한 대체수단으로 목적 달성) · B9 수량/방향 논리 모순
- B10 핵심 사유 부정 반전(결함 핵심 사유를 다른 기준 사유로 바꿔치기)

라벨 무결성(필수):
- 각 문장은 반드시 "잘못된 결함판단"이어야 한다(더 이상 결함 아님 / 틀린 기준 / 자체 모순).
  확신 없으면 그 축은 건너뛴다.
- B6/B7/B8은 보통 비결함이 되므로 "결함에 해당한다"가 거짓 → wrongnessReason에 명시.
- requirement에 근거. 헷갈리되 황당하지 않게.

출력: JSON 배열만. 각 원소
{"defectCase","statement","correctAnswer","trapAxis","changedFrom","changedTo","wrongnessReason"}
모든 사례에 대해(사례당 3개) 생성. JSON 외 산문 금지.
```
